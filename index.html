今日のルール：成果物の形式は、読み手で決める

AI駆動開発では、つい何でも Markdown（.md）で作らせがちです。しかし Markdown は開発者向けの形式です。クライアントや上司に .md ファイルを送りつけても読んでもらえません。

人が読むもの（要件書・論点リスト・分析レポート・設計書）は HTML で作らせます。ブラウザで開くだけで、整形された状態で読めます。
Bobへの入力になるもの（仕様書・再実行用の手順書）だけ Markdown にします。これはAIが読むファイルなので、それでいい。

この使い分けを、今日は最初から徹底します。

1-3矛盾を炙り出す ← 今日のハイライト

ここがコンサルの勝負どころです。RFPは、たいてい矛盾しています。それに初回訪問で気づけるかどうかで、信頼が決まります。

AGENT MODE
@lab1-rfp/01_RFP_工事原価管理システム刷新.pptx（本文）と @lab1-rfp/02_RFP別紙_要件一覧.xlsx（別紙）を突き合わせて、矛盾・不整合・確認が必要な点を洗い出し、issues.html として作成して。

【観点】
- 本文と別紙で、同じ要件について異なることを言っていないか
- 数値（性能・可用性・閾値・期日）の食い違いはないか
- スケジュールは、記載された工程を積み上げて物理的に成立するか
- 「必須」と書かれているのに、実現手段が矛盾している要件はないか

【出力形式】表形式で、列は以下：
No / 論点 / 本文の記述 / 別紙の記述 / なぜ問題か / 深刻度(高·中·低) / クライアントへの確認文

「クライアントへの確認文」は、そのまま質問票に転記できる丁寧な日本語で書いて。

【形式】
- 単一のHTMLファイル（CSS内包）。ブラウザで開くだけで読めること
- 深刻度「高」の行は視覚的に目立たせること
- 色は青とグレーのみ。フォントは IBM Plex Sans JP

答え合わせ：最低でもこの5つは出るはずです

論点矛盾の中身
会計連携先本文は「SAPとリアルタイム連携」、別紙は「弥生会計から月次CSV取込」。連携先も頻度も違う。ここを外すとアーキテクチャ全体が崩れる。
受注金額の正本文は「Salesforceがマスタ」、別紙は「見積書（確定版）が正」。マスタが2つある。Lab 3 でこれが実害を出していることが判明します。
実行環境本文は「パブリッククラウド可」、別紙は「国内オンプレミス必須（親会社方針）」。提案の前提が真逆。
スケジュール本文は「2027年4月1日 本番稼働（必達）」。別紙の工程を積み上げると 2028年9月。1年半足りない。
可用性本文「稼働率99.9%（計画停止含む）」。別紙「毎週日曜4時間の保守窓」＝年間約224時間停止 → 実質97.4%。要件を満たせない。

他にも5つ仕込んであります。全部見つけたら、あなたの勝ちです。

ここで立ち止まって考えてください。 Bob は「本文と別紙で書いてあることが違う」という事実を提示しました。でも、どれが本当に致命的かを決めたのはあなたです。会計連携先の矛盾は提案の骨格を壊しますが、モバイル対応の矛盾は後で調整できます。この重み付けは、AIにはできません。
  • issues.html が生成され、5つ以上の論点が挙がっている
  • そのうち「これは提案の前提を壊す」と自分で判断できたものが1つ以上ある
